# Forge v0.1 Design

**Date:** 2026-04-06

## Goal

Ship a final `v0.1` of The Forge that feels easy to launch, easy to test, and easy to demo. A user should be able to start the app from a single command, have the TUI handle orchestrator startup automatically, complete first-run setup inside the product, and optionally open a browser preview from the staging screen without needing to launch any extra app manually.

## Product Decisions

- `v0.1` is a hybrid release built on the `production-ready` worktree with selected `web-preview` features merged on top.
- The terminal TUI remains the only required control surface.
- The app must have one canonical startup command for users: `theforge`.
- The developer path may remain `go run ./BubbleTeaTerminal`, but it should behave like the released app as closely as possible.
- The Python orchestrator should never need to be started manually for normal use.
- The browser preview is opt-in polish from the staging screen, not part of the critical path.
- Documentation must describe one setup story, not separate legacy and release-era flows.

## Why This Shape

The two candidate worktrees solve different problems. `feat/production-ready` improves installation, setup, packaging, and runtime flexibility. `feat/web-preview` improves the staging and demo experience after an item has been generated. For a first public `v0.1`, shipping and startup reliability matter more than visual polish, but the browser preview is valuable if it can be added without complicating the launch model. The resulting design is therefore a hybrid: use `production-ready` as the release backbone and bring over the web preview as an optional staging enhancement.

## User Experience

### First Launch

The user installs or opens The Forge and runs a single command:

```bash
theforge
```

If required configuration is missing, the app opens a setup wizard instead of failing in the shell. The wizard should:

- choose cloud or local runtime mode
- collect the minimum required config for the chosen mode
- validate the values enough to prevent obvious dead-ends
- save a stable config file for later launches

### Normal Flow

After setup, the user flow is:

1. launch `theforge`
2. enter an item prompt
3. choose or confirm generation settings in the TUI
4. let the TUI auto-start or reconnect to the orchestrator
5. wait for generation to reach staging
6. choose one of the staging actions:
   - `Enter` to inject
   - `V` to open browser preview
   - `C` to craft again

### Testing and Demo Flow

The same startup story should work for demos and local verification. The browser preview exists to make staged output easier to inspect and present, but if it fails, the terminal preview and injection flow remain usable.

## Architecture

The final `v0.1` should combine two existing architectural directions without adding a new top-level app:

- `BubbleTeaTerminal` remains the product shell and entrypoint.
- The TUI owns orchestrator lifecycle management.
- The TUI also owns the optional preview server lifecycle.
- The Python agent runtime stays bundled with the app release.
- The mod bridge remains a separate tModLoader-side dependency, but the app should make that dependency explicit and easy to understand.

This keeps the app mentally simple: one product process the user starts, one pipeline the product manages, and one optional browser surface for richer preview.

## Core Components

### 1. Release Backbone

Adopt the `production-ready` branch as the base for:

- release artifacts for macOS and Windows
- post-install scripts
- stable config path and first-run setup
- support for cloud and local generation backends

This is the foundation of `v0.1` because it makes the app operable by more than just the current development workflow.

### 2. Unified Startup Path

Startup should be normalized across dev and release use:

- release users run `theforge`
- developers run `go run ./BubbleTeaTerminal`
- both paths should start the TUI first
- the TUI should launch the orchestrator in a predictable Python environment when needed
- startup failures should be surfaced inside the TUI with actionable guidance

The user should not need to know whether the orchestrator is a separate Python process.

### 3. Config and Environment Model

The app should prefer a product-level config file over ad hoc environment setup, while still allowing env overrides when needed for development or CI. `v0.1` should have one authoritative configuration story that explains:

- where config is stored
- what values are required per mode
- when `.env` is still relevant internally
- how local mode differs from cloud mode

### 4. Optional Browser Preview

Merge the `web-preview` flow as a staging-only enhancement:

- the staging screen exposes a `V` keybinding
- pressing `V` starts or reuses an in-process localhost preview server
- the preview page reads the current staged session and renders a category-specific scene
- the preview must never block generation, staging, or injection

This preserves the one-command model because the browser preview is served by the TUI rather than a separate frontend toolchain.

### 5. Docs and Install Story

The README and supporting docs must be rewritten around the final product narrative:

- install prerequisites
- first launch
- one-command startup
- optional browser preview
- ForgeConnector setup
- local vs cloud mode
- testing and smoke checks

The docs should stop teaching users to manually run the orchestrator unless they are doing explicit low-level debugging.

## Data Flow

1. User launches `theforge`.
2. TUI checks config state.
3. If config is incomplete, the setup wizard runs and persists config.
4. TUI ensures the orchestrator is running or starts it.
5. User submits a generation request.
6. Python agents execute the existing generation pipeline.
7. TUI polls status and reaches staging.
8. User may inject directly or open browser preview.
9. If browser preview is opened, the TUI publishes the current staged session to the in-process preview server.
10. Browser preview renders the current staged item without taking over the app flow.

## Error Handling

- If setup is incomplete, route to setup instead of failing in the terminal.
- If orchestrator startup fails, show a clear in-app error with the missing dependency or path problem.
- If the selected backend is unsupported on the machine, explain the fallback or the required installation step.
- If browser preview startup fails, keep the staging screen usable and show the preview URL only when available.
- If Terraria assets are unavailable for browser preview, fall back to placeholders with a visible warning.
- If ForgeConnector is not installed or the game is not running, injection should fail clearly without losing the staged result.

## Testing Strategy

### Automated

- Go tests for first-run setup, config loading, startup state transitions, and preview server integration
- Go tests for preview keybinding and session publishing behavior
- Python smoke tests for orchestrator startup and ready-state generation flow
- release workflow validation for archive contents and binary naming

### Manual

- fresh-machine startup simulation with no config present
- cloud-mode setup and generation pass
- local-mode setup and generation pass
- staging flow with and without browser preview
- injection flow with and without ForgeConnector available
- release artifact smoke test on macOS and Windows

## Success Criteria

- A new user can start the product from one command without manually launching the orchestrator.
- First-run setup happens inside the app and leads to a recoverable, repeatable config state.
- The TUI can generate, stage, and inject items through the normal flow.
- Browser preview can be opened from staging as an optional enhancement.
- Failure in browser preview does not block the main forge flow.
- The release documentation describes one clear path for installation and use.
